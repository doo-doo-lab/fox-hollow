## 一、核心开发纪律

### 1.1 遇事不决，参考猫国百科

**这是第一优先级规则。** 任何设计或实现上的不确定，**必须先查阅** `reference/` 目录下的猫国建设者（Kittens Game）中文 Wiki 百科：

- 机制不确定 → 查 `001-猫咪百科/`（建筑/村庄/科学/工坊/宗教/空间/时间）
- 资源平衡不确定 → 查 `003-资源大全/`（60+ 种资源详解）
- 数值公式不确定 → 查 `002-常用资料/`（游戏机制、价格指数、能源、数学技巧）
- 概念名词不确定 → 查 `005-名词解释/`（幸福度、收益递减等）
- 中后期节奏不确定 → 查 `000-猫国萌新指导/`（中期指导、自动重置、无限流等）

猫国是本项目的机制蓝本。虽然文案/命名/主题完全原创，但**底层玩法逻辑（资源链、建筑通胀、职业产出、政策不可逆、信仰转化公式等）应与猫国保持同级别的设计严谨度**。

### 1.2 兼容现有版本

开发新功能时，**必须**与游戏现有版本的以下方面保持兼容对接：

- **资源体系**：新资源必须接入 `calcR()`/`calcMx()` 计算链，不能孤立存在；新消耗必须有对应产出路径
- **建筑/研究解锁链**：新内容的 `uq`（解锁条件）必须接入现有 `chk()` 逻辑，不能断链
- **存档兼容**：任何新增 `G` 状态字段必须在 `migrate()` 中补齐默认值，保证旧存档不崩
- **UI 渲染**：新 Tab/面板必须适配现有 `rTC()` 渲染框架，不引入新的渲染模式
- **分支门控**：工业（`br:'I'`）/ 灵修（`br:'M'`）/ 副线（`sb:'D'`/`sb:'T'`）标记必须正确，防止内容泄漏到未选择的路线

### 1.3 注意逻辑错误

以下是历史上反复出现的逻辑错误类型，开发时必须主动检查：

- **速率口径不一致**：`data.js` 中定义的 `P` 值是每 tick 值（200ms），显示给玩家的是每秒值（×5），悬浮面板/描述文字必须统一为 `/s`
- **浮点精度**：资源值 < 0.0005 时应过滤为 0，防止显示 `-0.00`
- **freeFox 公式**：`foxes - foxAway - sum(jobs)`，卖建筑/远行归队/饥荒减员时都必须重算
- **政体正面/负面效果**：正面效果受令台加成，负面效果**不受**令台加成（v0.16.0 已修复过此 bug）
- **autoCraft 双计数**：连续速率模型是唯一来源，不可同时有离散批次执行（v0.13.2 已修复过）
- **离线补算**：`simulateOffline()` 必须覆盖所有时间推进系统（季节/远行/商队/节令/建筑维持/谷声累积）
- **阶段门控泄漏**：标记了 `phase` 或高前置条件的内容不应因中间研究完成而提前暴露

### 1.4 资源发展梯度规则

**资源体系必须呈现从低到高的发展梯度，不允许高阶内容使用低阶原料直接生产。**

核心原则：
- **每个阶段的建筑/配方/研究，其造价和消耗应以该阶段或上一阶段的资源为主**，不能跨越多个阶段回头大量使用最基础资源
- **高阶资源必须由中阶资源加工而来**，中阶由低阶加工而来，形成层层嵌套的资源链
- 基础资源（野莓/圆木/碎石）可以作为高阶内容的**微量辅料**，但**不能作为主要消耗**

具体约束：

| 阶段 | 主要造价/消耗资源 | 可用辅料 | 禁止 |
|------|------------------|---------|------|
| 第一纪（蛮荒） | 野莓、圆木、碎石 | — | — |
| 第二纪（聚落） | 木板、砖块、矿铁、铜钱 | 圆木、碎石（辅料） | 野莓作为建筑主材 |
| 第三纪（文化） | 染丝、果酒、墨锭、丝帛、香草 | 铜钱、卷轴（辅料） | 圆木/碎石作为主材 |
| 第四纪（立政） | 卷轴、铜钱、古币、谷声门槛 | — | 基础资源作为政治花费 |
| 第五纪·工业 | 煤→钢→火油→寒钛→辉石→重晶 | 矿铁（辅料） | 圆木/碎石/木板作为工业主材 |
| 第五纪·灵修 | 灵能→命丝→共振子→晶丝→元念 | 符咒（辅料） | 基础资源作为灵修主材 |
| 第六纪（终局） | 各线终局专属资源 | 高阶中间材料 | 任何基础/中阶资源作为终局主材 |

**参照猫国**：猫国的资源链严格遵循 猫薄荷→木→矿→铁→煤→钢→钛→合金→铀→反物质 的层层递进，后期建筑绝不会用猫薄荷或木材作为主要造价。本游戏必须保持同等水平的资源梯度设计。

**检查方法**：新增建筑/配方时，对照上表检查其造价中的资源是否属于当前阶段±1层。如果出现跨2层以上的资源作为主材（如工业线建筑用碎石做主造价），属于设计错误，必须修正。

### 1.5 不确定就停下来问

**查了猫国百科仍然不确定的问题，不要擅自决定。** 停下来问策划（我）。

以下情况必须停下来确认：
- 新资源的**命名**（所有命名需策划审核，避免撞猫国专属术语）
- 新机制的**数值**（产出速率、消耗量、概率、上限）
- **不可逆选择**的具体效果和代价
- 影响**多个系统**的交叉改动（如政体效果同时影响 calcR/calcH/foxEatRate/bp）
- **删除或重构**现有系统
- 任何**叙事文本**内容（短句/事件/碎片/日志）

## 五、开发约定

1. **纯前端项目**：HTML + CSS + JS，不引入框架和构建工具。
2. **文件分工明确**：
   - `data.js` — 所有静态数据定义（RD/BD/JD/UD/CD/UPGD/SD/ED/WD/POLITY/POLICY/EXD/NARR/CVD/SPEC_BD/SPEC_JD/CUSTOMS 等）
   - `engine.js` — 引擎核心（G 状态、计算函数 calcR/calcMx/calcH/calcEnergy/calcLeyline/calcPollution/calcUnrest、tick 循环、离线补算、事件触发、存档）
   - `engine-actions.js` — 玩家直接触发的操作（采集/建造/研究/制作/分配/灵术/商队/政体/政策/习俗/节令）
   - `engine-systems.js` — 自动子系统（远行/抉择事件/商队生成/图纸抽取）
   - `ui.js` — 界面渲染（Tab/面板/悬浮/modal/日志）
   - 不跨文件混写逻辑。
3. **叙事文本风格**：冷幽默、有哲思、克制、口语化。短句（tip）不超过两句，不用感叹号。
4. **短句默认不写**：新增内容默认不写 tip 短句，留空或占位。需要写时由策划明确指示。
5. **存档兼容**：任何新增状态字段必须在 `migrate()` 中补齐默认值，保证旧存档不崩。
6. **提交信息**：中文，简明扼要，说清改了什么、为什么改。
7. **分支标记**：工业线内容标 `br:'I'`，灵修线标 `br:'M'`，副线标 `sb:'D'`（神启）/ `sb:'T'`（通达）。未标记的为共通内容。
8. **命名禁区**：避开猫国专属术语（猫薄荷、独角兽、天角兽、死灵兽、卡米、天网、虚空裂隙等）。所有命名原创。
9. **缓存版本号必须递增**：修改 `js/*.js` 后，必须在 `index.html` 中递增对应的 `?v=N` 缓存版本号（如 `<script src="js/ui.js?v=16">` → `?v=17`）。否则浏览器会一直用缓存的旧版本，玩家看到错误的代码。当前版本号位于 [index.html](../index.html) 行 47-51。规则：
   - 改 `js/data.js` → 递增 `data.js?v=N`
   - 改 `js/engine.js` → 递增 `engine.js?v=N`
   - 改 `js/engine-actions.js` → 递增 `engine-actions.js?v=N`
   - 改 `js/engine-systems.js` → 递增 `engine-systems.js?v=N`
   - 改 `js/ui.js` → 递增 `ui.js?v=N`
   - 同 commit 内多次改同一文件，递增一次即可。

2. **不为单一机制新建文档**——写进对应 `branch-*.md`
3. **只在新增主线/副线时**才新建 `branch-*.md`
4. **禁止创建 `v0.X-plan.md` / `*-stub.md` / `*-rewrite.md` / `*-v2.md` 形式文档**——这类命名暗示"临时/迭代/未定稿"，容易过期混乱。roadmap-v2 §对应章节足够承载阶段性方案
5. **临时方案 / 草案**——写在 commit message 或 PR 描述中，不进 `docs/`
6. **触发归档**——以下任一条件即移至 `plan/archived/` 并更新 `archived/README.md`，**不直接删除**：
   - 设计被新方案完全替代
   - 计划已实施完成且 changelog 已记录
   - 提案不再考虑实施

### 7.3 防混乱 4 原则

| 原则 | 含义 | 落地方法 |
|------|------|---------|
| **单一信息源** | 每个事实只在一处记录 | 多处提及用 `[link](path)`，不复制内容 |
| **状态明确** | 每段设计标"状态：占位 / 设计中 / 待实施 / 已实施" | 占位状态才写"待补完清单"，已实施状态移除清单 |
| **勘误集中** | 跨文档不一致汇总到 `roadmap-v2.md` §十五 表格 | 不散落在各文档脚注 |
| **AI 入口固定** | `docs/plan/INDEX.md` 是 AI 第一站 | 进入项目先读 INDEX，按需展开 |

### 7.4 AI 工作流程（修改 docs/plan/ 时严格遵守）

1. 先读 `docs/plan/INDEX.md` 摸全局结构和 Phase 索引
2. 用 `Grep "^##"` 摸具体文档大纲
3. 用 `Read offset/limit` 读相关章节
4. 修改时同步检查是否需要更新：
   - `roadmap-v2.md` §十五 勘误（发现跨文档不一致时）
   - `INDEX.md`（文档结构变化时）
   - `changelog.md`（版本级变更时）
   - 其他 branch（如改了某个共通系统的命名）

### 7.5 状态标签

设计章节首部统一标 `状态：` 字段，可选：

- **占位**——框架已搭，数值/细节未定，必须包含"开发前待补完清单"
- **设计中**——正在迭代，可能有多版本草案
- **待实施**——设计定稿，等开发周期排期
- **已实施**——代码已落地（同步 `changelog.md` 版本记录）

由"待实施"→"已实施"时**移除"待补完清单"**，并在 `changelog.md` 加版本记录。

### 7.6 文件命名规范

- `RULES.md`, `design.md`, `changelog.md`——根级核心文档，名称固定
- `roadmap-v2.md`——路线图主入口（如未来需大改，命名为 `roadmap-v3.md` 并把 v2 归档）
- `branch-{name}.md`——主/副分支详设（每条线一个）
- `branch-{parent}-{child}.md`——分叉子文档（如 `branch-divine-doctrine`）
- `archived/{原名}.md`——归档不改名，保留辨识度

### 7.7 设计冲突解决优先级

跨文档信息不一致时，权威顺序（高→低）：

1. **branch-{name}.md 设计表**（如 §六全建筑表 / §十四全职业表）——分支设计的最终权威数据，作为后续开发的目标
2. **roadmap-v2.md 主体章节**——全局阶段映射、副线门控、共通系统公式（神恩/声望等）
3. **代码现状**（`js/data.js` 等）——**已实施且无 rename 计划**的部分以代码为准；若与设计表冲突，见 §7.8 双轨规则
4. **branch 文档中的"实现记录" / "过时警告"段**——反映某次施工的代码状态，**不是权威数据源**，仅供对比参考
5. **roadmap-v2 §十五 勘误**——已发现并记录的不一致，按勘误的"修正"列处理
6. **design.md**——历史回顾、已完成内容描述
7. **archived/**——历史参考，不作权威

发现冲突时：**修高优先级文档以匹配实际，并在 roadmap-v2 §十五 加一条勘误记录**。

### 7.8 代码 ID 与设计目标 ID 不一致（双轨规则）

当代码已上线但设计想 rename 时（如代码 `spiritChanneler` → 设计 `spiritSenser`，见 [E48](plan/roadmap-v2.md)）：

1. 在 branch-{name}.md 顶部加 **"过时警告" 块**，显式列出差异（参见 [branch-mystic.md 行 56](plan/branch-mystic.md)）
2. **设计章节使用目标 ID**（spiritSenser）——这是后续开发的权威
3. **"实现记录"章节使用代码现状 ID**（spiritChanneler）——这是当前代码的诚实记录
4. roadmap-v2 §十五 加勘误条记录 rename 计划及目标 Phase
5. **rename 落地时**（在某个 Phase 完成 migrate 后）：
   - 删除"过时警告"中对应行
   - 把"实现记录"改成目标 ID
   - changelog 加 rename 记录
   - roadmap-v2 §十五 勘误条标 ✅ 已修正

**AI 读到代码与设计 ID 不同时**：先查 branch-{name}.md 顶部的"过时警告"，按其指向判定权威。无过时警告时，按 §7.7 优先级（设计表 > 代码）。

## 八、常见错误与 mulerun 限制

> 本节为 mulerun（接手开发的 AI）必读。**违反任一条 = 错误**。
> 完整开发 SOP 见 [docs/plan/DEV_SOP.md](plan/DEV_SOP.md)。

### 8.1 mulerun 两条硬限制

| # | 限制 | 违反后果 |
|---|------|---------|
| 1 | **不得主动 git push**：完成 commit 后停下报告"已 commit，等待 review"，由人类决定何时 push | 错误代码可能进入远程，影响其他贡献者 |
| 2 | **不得跨步骤跳跃**：必须按 [roadmap-v2.md §四 实施进度表](plan/roadmap-v2.md) 中 ⏳ 的下一步做，禁止跳到后面的步骤 | 留半残状态、依赖错位、build 不通 |

> **关于修改设计文档**：mulerun 在**必要时**可以修改 roadmap-v2.md / branch-*.md 的设计内容（如发现冲突、勘误、或开发中需要细化设计）。但必须：
> 1. 在 commit message 明确说明改了什么 + 原因
> 2. 优先使用 [roadmap-v2.md §十五 勘误表](plan/roadmap-v2.md) 记录跨文档不一致
> 3. 改动应保留在最小必要范围（不顺手"改进"无关内容）
> 4. 改设计文档的同时必须更新进度表与 changelog（如适用）

### 8.2 mulerun 必须沉默地停下报告的 6 个场景

遇到以下情况，**立刻停下报告**，不要 silently 选择方向继续：

1. **任务卡前置未满足**：依赖的步骤未标 ✅
2. **文档与代码冲突**：branch-*.md 设计表 vs data.js 实际值不一致，且无"过时警告"
3. **验收清单某项 fail**：任何 `[ ]` 项验证不过
4. **console 出现错误**：preview_console_logs 显示任何 error
5. **migrate 后旧存档崩**：测试存档加载失败
6. **改动范围超出任务卡"涉及文件清单"**：动了不该动的文件

报告格式见 [DEV_SOP.md §四.3](plan/DEV_SOP.md)。

### 8.3 五大常见错误（按发生频率排序）

| # | 错误 | 90% 解法 | 预防 |
|---|------|---------|------|
| 1 | 改了 JS 但浏览器没更新 | 递增 [index.html](../index.html) 的 `?v=N` | 见 §五.9 cache-buster 规则 |
| 2 | 直接拿"代码现状"当权威 ID | 看 branch-*.md 顶部"过时警告" | 见 §七.7/七.8 优先级 |
| 3 | 半残状态（删 RD.council 但漏改 ui.js 引用）| 改某资源/字段时，全局 grep 该名字找所有引用 | 任务卡"涉及文件清单"必须完整 |
| 4 | migrate 漏加新字段，旧存档崩 | 测试时用旧版本存档码加载，看 console | 任何新 G.x 字段必须在 migrate 加 `G.x = G.x || ...` |
| 5 | commit 范围过大（混入意外文件）| `git status` + `git diff --stat` 在 commit 前检查 | 用 `git add <具体文件>` 而非 `git add .` |

### 8.4 何时 commit 与不 commit

**应该 commit 的场景**：
- 任务卡的所有 `[ ]` 验收清单全过
- 改动范围与"涉及文件清单"一致
- console 无错误
- 旧存档加载正常

**不应 commit 的场景**：
- 任何验收 fail
- 改动包含意外文件（如 .gitignore 删除）
- 浏览器还没 reload 验证
- cache-buster 没递增（如改了 JS）

### 8.5 commit message 必须包含的 4 个要素

参照 [DEV_SOP.md §三.3](plan/DEV_SOP.md)：

1. **scope: 章节号 摘要** 标题（如 `engine: §四 1.3 POLITY 不可逆化`）
2. **改动列表**（按文件分组）
3. **验证清单**（全打 ✅）
4. **cache-buster 版本号**（如适用）
5. **Co-Authored-By 行**（如使用 Claude）
