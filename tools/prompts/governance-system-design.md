# 任务：为《狐狸谷》典制系统做名字润色 + 数值对称性 + UI 简化建议

你是资深独立游戏设计师，熟悉放置类增量游戏（猫国建设者风格）的政体/政策系统设计。

为放置类增量游戏《狐狸谷》（一群狐狸在山谷里建文明）的"典制"页签做 4 项审视：

1. **6 政体名字润色**——现名"野风志/散爪活/合嗓令"等不够生动
2. **POLICY 16 个 opts 名字润色**——同上
3. **6 政体效果对称性检查 + 数值平衡建议**——内守 3 个 vs 外拓 3 个的强度是否平衡
4. **典制页签 UI 简化建议**——参考猫国信息密度

---

## 一、项目背景

- 名称：《狐狸谷》（Fox Hollow Tales）
- 类型：浏览器纯前端增量放置游戏，HTML+CSS+JS 无框架
- 主题：一群狐狸在山谷里从采野莓开始建文明
- 文风：长句白描 + 物件具体 + 不写"心情很好"这种总结尾巴
- 视觉：白底黑字纯文本，无图标，参考猫国建设者
- 玩家代词硬规则：所有狐狸**只用"它"**，禁"他/她/他们"

## 四、4 个具体任务

### 任务 1：6 政体名字润色

现状："守尾制 / 闭谷修灵 / 合嗓令 / 通货集 / 野风志 / 散爪活"——多数生硬，玩家觉得"不够有意思"。

**约束**：
- 保留 ID（elder/hermit/public/trade/martial/anarchy）不变
- 保留 tier1 分组（in vs out）
- 保留每个政体的核心政治隐喻（长老/隐士/公议 vs 商业/武力/无政府）
- 名字字数：2-4 字（短而有力）
- 风格：比"野风志/散爪活"更生动、更带画面感
- 保留 d 描述字段，可同步润色

**Tier 1 路线名"内守 / 外拓"**也可以一并润色（如果你觉得不够生动）。

### 任务 2：POLICY 16 opts 名字润色

现状：公田/私田/共耕、师徒/书院/自学、开放通商/管控/封闭、平等/长幼有序/才能至上、通达/闭谷。

部分还可以（"公田/师徒/平等"），但有些需要润色（"封闭/合嗓令/管控"）。

**约束**：
- 保留 opt ID
- 保留每个 opt 的核心机制（产出/成本/惩罚类型）
- 字数 2-4 字
- 跟政体名字风格统一

### 任务 3：6 政体效果对称性 + 数值平衡

玩家观察："内守"和"外拓"的奖励加成相当吗？

**当前主要数值统计**：

内守 3 个：
- elder: hapM+8% loreM+50% / caravanProb-5% expReward-10%
- hermit: charmM+15% loreM+10% spellEff+10% / coinM-10% caravanProb-5% diplomatRes-15%
- public: hapM+12% allM+3% policyCost-30% / buildCost+5% researchCost+5%

外拓 3 个：
- trade: coinM+20% caravanProb+8% bpChance+8% / hapM-5% baseProd-3%
- martial: expReward+15% baseProd+10% scout+10% / hapM-8% coinM-5%
- anarchy: gatherM+30% jobM+8% hapCap+20% / bldProd-10% researchSpeed-5%

**任务**：
- 评估每个政体"正面强度 - 惩罚强度"的近似分值
- 找出明显偏强（OP）/ 偏弱（垃圾）的政体
- 内守 3 个 vs 外拓 3 个 整体比较
- 给出**具体数值调整建议**（比如"elder loreM +50% 太离谱，建议 +25%"）

**约束**：
- 数值改动控制在 ±50%（不要重写整个系统）
- 保留每个政体的"特色机制"（elder = 学识王、anarchy = 手动采集王 等）
- 输出可直接抄回代码的 e/pen 字段（保留 key 名）

### 任务 4：典制页签 UI 简化建议

当前典制页签（旧截图描述）：
- 顶部：本季节令（彩络+醴浆+墨锭，三全礼）
- 中间：已激活习俗 9/10，习俗卡片网格（每张 ~150px 宽，含名字/描述/状态/激活按钮/tip）
- 底部：治理面板（路线 2 卡 / 政体 6 卡 / 5 个政策域）

玩家反馈："典制的 UI 界面不够简化，不够好看。"

参考猫国建设者的信息密度：
- 行高低、间距小
- 文字密集，按钮小
- 多列布局
- 状态用图标/颜色区分（绿=激活，灰=未解锁，黑=可激活）

**任务**：
- 给具体的 UI 简化建议（5-10 条）
- 每条带"现状/改后/原因"
- 优先 CSS 改动，避免大改 HTML 结构
- 保留所有功能（习俗激活/政体选择/政策选择）
- 不必全实现，给方向就行
